## 1. תקלה קריטית — איפוס סיסמה תקוע על "מאמת קישור..."

**תסמין:** מנהל המכללה לחץ על קישור איפוס מה-WhatsApp. מסך האיפוס נותר תקוע בהודעה "מאמת קישור..." ללא מעבר.

**סיבת שורש:**
Supabase SDK עבד במצב **PKCE** (ברירת מחדל). ב-PKCE, בעת בקשת איפוס, נשמר `code_verifier` ב-`localStorage` של הדפדפן שביקש. כשהמשתמש פותח את הקישור בדפדפן **אחר** — למשל דפדפן פנימי של WhatsApp, Telegram, Gmail, Outlook — ה-verifier לא זמין שם. Supabase נכשל בהחלפת ה-code ל-session בשקט, ו-`getSession()` מחזיר `null` לנצח. ה-polling שבודק שה-session מוכן רץ עד אין סוף.

**פתרון יציב:**
- `src/supabaseClient.js`: מעבר ל-`flowType: 'implicit'`. ה-`access_token` נכלל ישירות ב-URL hash של קישור האיפוס — לא דרוש state מקומי, עובד בכל דפדפן.
- `src/components/PublicForm.jsx`: timeout של 8 שניות על ה-polling, עם הודעת שגיאה ברורה: *"הקישור לא זמין בדפדפן הזה. פתח/י את הקישור ישירות ב-Chrome/Safari, או בקש/י קישור חדש."*

**למה זה יציב:** ה-implicit flow הוא התקן הישן של OAuth ופועל על כל דפדפן/פלטפורמה. ה-timeout מבטיח שגם במקרי-קצה שנפספסו (קישור פג-תוקף, חסום רשתית), המשתמש יקבל הנחיה במקום להיתקע.

## 2. תקלה קריטית — "המשתמש לא נמצא במערכת" אחרי שינוי מייל ע"י אדמין

**תסמין:** האדמין החליף מייל של איש צוות (מייק דורון → `mikedoro@gmail.com`). מייק עשה איפוס סיסמה מוצלח, אך בכניסה ראה "המשתמש לא נמצא במערכת".

**סיבת שורש:**
מסלול ה-`/api/staff action=update` עדכן רק את טבלת `staff_members.email`, בלי לסנכרן את `auth.users.email` ו-`public.users.email`. נוצרה "סכיזופרניה" בנתונים: Supabase Auth זיהה את המשתמש עם המייל הישן, אבל טבלת ה-staff אחזקה את החדש. כאשר הכניסה נכשלה, הלוגיקה ב-`routeByRolesCore` גם לא בדקה כלל את טבלת `staff_members` לפני ההודעה "לא נמצא" (רק סטודנטים ומרצים).

**פתרון יציב:**
- `api/staff.js` (`action=update`): לפני ה-PATCH נשמר המייל הישן; לאחר מכן מחפשים את ה-auth user גם לפי הישן וגם לפי החדש, מעדכנים את `auth.users.email` (עם `email_confirm=true` כדי לוותר על מייל אישור נוסף) ומשלימים mirror ב-`public.users`.
- `api/staff.js` (`handleUpdateUser`, המסלול החדש): מסנכרן מייל ל-auth באותו אופן.
- `src/components/PublicForm.jsx` (`routeByRolesCore`): כשאין שורת `public.users`, בודק גם את `staff_members` לפני ההודעה השלילית.

**למה זה יציב:** העדכון עכשיו אטומי מבחינת המשתמש — כל נקודות ה-truth (auth.users, public.users, staff_members) מתיישרות. הגילוי של ה-auth user לפי המייל הישן שורד גם מצבי drift.

## 3. שיפור ביצועים — חיפוש auth user לפי מייל

**הבעיה (נמצאה במהלך הסקירה):**
`findAuthUserByEmail` מעמוד דרך Admin API של Supabase, 1000 משתמשים בעמוד, עד 50 עמודים. בקנה-מידה גדול (אלפי משתמשים), כל פעולת אדמין שמזהה משתמש (שינוי מייל, עריכת שם) יכולה לקחת שניות ארוכות.

**פתרון יציב:**
נוסף **fast path**: חיפוש ב-`public.users` (אינדקס טבעי על העמודה `email`) → שליפת ה-auth user לפי `id` בבקשה אחת. O(1) במקום O(n). ה-fallback הישן נשמר למקרי drift.

## 4. מסלולים שנבדקו ונמצאו תקינים

- **Service Worker / PWA:** `skipWaiting`, `controllerchange` + reload עם safety של 12 שניות, `cleanupOutdatedCaches` — אין בעיית cache.
- **ניתוב כפול session + recovery:** `recoveryModeRef` משמש guard כפול; סקריפט inline ב-`index.html` מנקה `sb-*-auth-token` לפני שה-SDK נטען כדי למנוע route מוקדם לפורטל.
- **Timeout חוסר פעילות:** 30 דקות — קיים וסביר.
- **הודעת "אימייל או סיסמה שגויים":** נשארת כללית במכוון (standard anti-enumeration) — לא נחשוב על הבדל בין "אין מייל" ל"סיסמה שגויה" בשלב ה-login.
